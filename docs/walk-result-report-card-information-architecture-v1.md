# Walk Result Report Card Information Architecture v1

## 연결 이슈
- #702
- #267
- #266

## 목적
- 산책 종료 직후와 산책 상세 화면이 `같은 결과를 다른 밀도로 읽는 구조`가 되게 만든다.
- 사용자가 가장 먼저 궁금해하는 질문 3개만 빠르게 답하고, 나머지는 펼쳐보기 상세로 내린다.
- 내부 계산 용어가 그대로 튀어나오지 않게 하면서도, 상세 근거를 잃지 않게 만든다.

## 제품 원칙
- 구조는 `짧은 요약 + 펼쳐보기 상세`를 유지한다.
- 종료 직후는 해석보다 이해 속도를 우선한다.
- 상세 화면은 근거를 확장하되 큰 장문 카드 하나로 밀어 넣지 않는다.
- 작은 화면에서도 카드 높이가 과도하게 커지지 않도록 한 화면에서 보이는 텍스트 블록 수를 제한한다.
- 종료 직후와 상세 화면이 다른 제품처럼 보이지 않도록 같은 상태명과 같은 질문 순서를 쓴다.

## 사용자가 먼저 답 받아야 할 질문
1. 이번 산책에서 얼마나 반영됐는가
2. 왜 일부는 제외됐는가
3. 어디에 이어지는가

## Surface 역할 분리
### 1. 종료 직후 요약
Canonical surface:
- `map.walk.savedOutcome.card`

역할:
- 사용자가 저장 직후 결과를 3초 안에 이해하게 한다.
- 숫자 전체를 다 보여주지 않고, 핵심 상태와 한 줄 근거만 보여준다.
- 다음 행동은 `목록에서 보기` 같은 재진입 CTA 하나로 제한한다.

카드 구조:
1. 상태 헤드라인
2. 이번 산책 반영 요약 1줄
3. 제외가 있다면 최상위 이유 1줄
4. 어디에 이어지는지 1줄
5. CTA 1개

높이 규칙:
- 본문 블록은 최대 4개
- 각 문장은 2줄 이내
- 제외 이유는 최대 1개만 직접 노출
- `mark`, `route`, `감쇠`, `cap` 같은 내부 항목은 종료 직후 화면에서 숨긴다

### 2. 종료 전후 연결 설명 seed
Existing support surface:
- `walk.detail.valueFlow.card`

역할:
- 최종 저장 전 사용자가 `기록 / 다시 볼 곳 / 이어지는 결과`를 먼저 이해하게 만든다.
- 결과 리포트의 상세를 전부 담는 카드가 아니라, 후속 상세 화면을 읽는 준비 카드로 둔다.

규칙:
- 저장 전에는 계산 디테일을 길게 보여주지 않는다.
- 종료 직후 요약과 같은 용어를 사용하되, 세부 수치 분해는 하지 않는다.

### 3. 저장된 산책 상세
Canonical anchor:
- `walklist.detail.loopSummary`

상세 화면에서는 기존 hero 요약 아래에 `결과 설명 리포트 섹션`이 이어진다고 가정한다.

상세 섹션 순서:
1. 상태 헤드라인 요약
2. 반영/제외 수치 요약
3. 제외 사유 펼침 섹션
4. 연결 흐름 펼침 섹션
5. 계산 근거 요약 섹션

## 상세 화면 정보 구조
### A. 상태 요약 섹션
표시 값:
- 상태명
- 반영 포인트 수
- 제외 포인트 수
- 제외 비율

이 섹션은 항상 접지 않는다.

### B. 제외 사유 섹션
기본 상태:
- 접힘

표시 규칙:
- 최대 3개 reason row
- 각 row는 `사유명 + count + 짧은 설명`만 노출
- 상세 화면에서도 긴 정책 문단은 허용하지 않는다

CTA:
- `제외 이유 보기`
- `제외 이유 접기`

### C. 연결 흐름 섹션
표시 순서:
1. 기록
2. 영역/목표
3. 시즌
4. 미션/퀘스트

각 row는 아래 3가지만 유지한다.
- 연결 대상명
- 상태 배지
- 1줄 설명

### D. 계산 근거 섹션
종료 직후에는 숨기고, 상세에서만 노출한다.

표시 항목:
- 영역 표시 기여
- 경로 기여
- 감쇠 적용
- 상한 적용

노출 원칙:
- 내부 raw key는 숨긴다
- 숫자와 번역된 라벨만 보여준다
- 이 섹션이 summary보다 먼저 올라오지 않는다

## 상태별 톤 규칙
### 거의 반영 안 됨
- 종료 직후: `이번 산책은 저장됐지만 실제 반영은 적었어요.`
- 상세: 제외 비율과 주요 사유를 먼저 보여준다.
- 강조 순서: 제외 이유 -> 다시 확인할 화면

### 정상 반영
- 종료 직후: `이번 산책이 정상 반영됐어요.`
- 상세: 연결 흐름을 먼저 보여주고, 제외는 보조 정보로 둔다.
- 강조 순서: 반영량 -> 어디에 이어지는가

### 정책 제외 다수
- 종료 직후: `보호 기준 때문에 일부 기록이 많이 제외됐어요.`
- 상세: 정책 reason row를 맨 위에 둔다.
- 강조 순서: 이유 -> 연결 영향 -> 계산 근거

## 카피 규칙
- 내부 용어 노출 금지
  - `mark`
  - `route`
  - `decay`
  - `cap`
  - `fallback`
- 사용자는 해석된 라벨만 본다
  - `영역 표시 기여`
  - `경로 기여`
  - `감쇠 적용`
  - `상한 적용`

## 밀도 규칙
- 종료 직후 화면은 단락형 설명문을 금지한다.
- 상세 화면도 섹션당 3개 row를 넘기지 않는다.
- 320pt급 작은 폭에서도 첫 화면은 상태 요약과 CTA가 모두 보여야 한다.
- 긴 설명이 필요하면 새 문단이 아니라 펼침 섹션으로 보낸다.

## DTO 연결 규칙
- 종료 직후와 상세 화면은 같은 `WalkOutcomeExplanationDTO`를 읽는다.
- 종료 직후는 `summaryState`, `appliedPointCount`, `excludedRatio`, `topExclusionReasons.first`, `connections.primary`만 사용한다.
- 상세 화면은 same DTO 전체를 사용한다.

## 구현 가이드
- 종료 직후 surface는 기존 `map.walk.savedOutcome.card`를 결과 요약형으로 진화시키는 방향이 우선이다.
- 저장된 산책 상세는 `walklist.detail.loopSummary`를 hero anchor로 유지하고, 그 아래에 결과 설명 리포트 섹션을 추가한다.
- `walk.detail.valueFlow.card`는 저장 전후 연결 설명 seed로 유지하고, 상세 리포트 전체 책임을 떠안기지 않는다.
