# Privacy Deletion Request Intake Tracking v1

- Issue: #720
- Relates to: #268, #700, #704

## 목표
- 삭제 요청을 `mailto 링크 한 번 열기`가 아니라 `요청 ID가 있는 접수 흐름`으로 정리한다.
- 설정 탭, 사용자 copy, 운영 응답 기준이 같은 단계 이름을 쓰도록 고정한다.
- 일반 지원 문의와 삭제 요청을 서로 다른 액션으로 분리한다.

## 사용자 표면 원칙
1. 삭제 요청은 프라이버시 센터의 전용 카드/시트로 연다.
2. 일반 문의 메일과 같은 entry로 보이지 않게 분리한다.
3. 메일을 보내기 전 무엇이 함께 전송되는지 먼저 보여준다.
4. 전송 뒤에는 `다음에 무엇이 일어나는지`를 앱 안에서 바로 설명한다.
5. 상태 문의는 항상 같은 요청 ID를 재사용한다.

## 삭제 요청 canonical 단계

### 1. 접수 전
- badge: `접수 전`
- 의미:
  - 아직 요청 ID가 없다.
  - 사용자에게 수집 항목과 다음 단계를 먼저 보여준다.

### 2. 초안 준비됨
- badge: `초안 준비됨`
- 의미:
  - 요청 ID를 발급했다.
  - 메일 전송은 아직 확인하지 못했다.

### 3. 전송 확인 대기
- badge: `전송 확인 대기`
- 의미:
  - 외부 메일 앱으로 handoff했다.
  - 앱은 실제 send 완료를 직접 알 수 없으므로, 사용자가 전송 완료 여부를 확인해야 한다.

### 4. 회신 대기
- badge: `회신 대기`
- 의미:
  - 메일 전송을 확인했다.
  - 운영팀의 첫 접수 회신을 기다리는 상태다.

### 5. 문의 준비됨
- badge: `문의 준비됨`
- 의미:
  - 같은 요청 ID로 상태 문의 초안을 다시 만들었다.
  - 회신은 기존 스레드와 같은 요청 ID로 이어진다.

## 요청 ID 규칙
- 형식: `DEL-YYYYMMDD-HHMMSS-XXXXXX`
- 용도:
  - 메일 제목
  - 메일 본문
  - 앱의 현재 상태 카드
  - 상태 문의 재진입
- 한 건의 삭제 요청은 처리 완료 전까지 같은 요청 ID를 유지한다.

## 사용자에게 먼저 보여줄 수집 항목
- 요청 ID
- 현재 계정 이메일 또는 `guest`
- 앱 버전
- 빌드 번호
- 번들 ID

이 5개는 삭제 요청 메일 본문에 항상 포함한다.

## 메일 작성 정책

### 1. 우선 경로
- `MFMailComposeViewController`를 우선 사용한다.
- 이유:
  - 앱 안에서 send / save / cancel 결과를 구분할 수 있다.
  - 더 이상 `mailto only`에 의존하지 않는다.

### 2. fallback 경로
- 기기에 Mail 계정이 없어 composer를 쓸 수 없으면:
  - `mailto:` fallback
  - 요청 ID 복사
  - 메일 본문 복사
- fallback에서는 앱이 실제 send 완료를 알 수 없으므로 상태를 `전송 확인 대기`로 둔다.

## 운영 intake 규칙
- 수신 채널: support mailbox
- 메일 제목 규칙:
  - 삭제 요청: `[DogArea 삭제요청][<REQUEST_ID>] 개인정보 삭제 요청`
  - 상태 문의: `[DogArea 삭제요청][<REQUEST_ID>] 처리 상태 문의`
- 일반 지원 문의 제목에는 삭제 요청 prefix를 쓰지 않는다.

## 운영 응답 SLA
- 1차 접수 회신 목표: 24시간 이내
- 추가 확인이 필요한 경우:
  - 같은 스레드에서 요청 ID를 유지한 채 필요한 정보를 다시 요청한다.
- 사용자가 상태 문의를 보내면:
  - 같은 요청 ID로 이어서 회신한다.

## 앱 안에서 설명해야 할 다음 단계
- `메일 전송 후 앱은 요청 ID와 접수 대기 상태를 보관합니다.`
- `운영팀은 같은 요청 ID로 24시간 안에 첫 회신을 보냅니다.`
- `추가 확인이 필요하면 같은 요청 ID로 상태 문의를 다시 보낼 수 있습니다.`

## 일반 문의와 삭제 요청 분리 규칙
- 설정의 `지원 / 문의` 섹션:
  - 제품 지원/버그/저장소 링크만 다룬다.
- 프라이버시 센터:
  - 삭제 요청
  - 보존기간 안내
  - 삭제 요청 처리 기준
  - 개인정보처리방침
- 두 경로는 제목, badge, request id 유무가 명확히 달라야 한다.

## 완료 판단 기준
- 프라이버시 센터에서 삭제 요청이 전용 card + sheet 흐름으로 보인다.
- 사용자가 메일 전송 전 어떤 정보가 수집되는지 말할 수 있다.
- 요청 후 어떤 회신을 기다려야 하는지 설명할 수 있다.
- 상태 문의가 같은 요청 ID를 재사용하는 규칙으로 정리돼 있다.
