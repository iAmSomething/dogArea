# Privacy Control Recovery Retention Flow v1

- Issue: #704
- Relates to: #268, #266

## 목표
- 공유 상태 제어, 권한 복구, 최근 상태 확인, 보존/삭제 요청을 각각 구현 가능한 사용자 플로우로 분해한다.
- `즉시 꺼졌다`, `아직 서버 반영 중이다`, `권한이 없어 다시 켤 수 없다`를 사용자가 혼동하지 않게 한다.

## 기본 상태 모델
- `guest`
- `member_private`
- `member_sharing_on`
- `member_permission_blocked`
- `member_offline_pending`
- `member_server_delayed`

## 공통 피드백 규칙
1. 상태 변경 버튼을 누르면 즉시 로컬 상태 문구가 바뀌어야 한다.
2. 서버 반영이 늦을 수 있으면 `반영 중`임을 따로 말한다.
3. 오류/오프라인이면 마지막 성공 상태와 현재 요청 상태를 구분해 보여준다.
4. 회원/비회원 상태 차이는 CTA 수준에서 분리한다.

## Flow A. 세션 단위 즉시 비공개

### 트리거
- 프라이버시 센터 `지금 바로 비공개`
- 라이벌/지도 shortcut에서 `공유 끄기`

### 사용자 기대
- 지금 산책 중이어도 더 이상 공유에 반영되지 않는다고 느껴야 한다.

### 상태 전이
1. `member_sharing_on`
2. 로컬 상태 즉시 `비공개로 전환 중`
3. presence 송신 중단
4. 성공 시 `member_private`
5. 실패 시 `member_server_delayed` 또는 `member_offline_pending`

### 성공 피드백
- 토스트: `지금부터 비공개예요`
- 보조 문구: `이미 반영된 익명 집계는 잠시 남을 수 있지만 새 공유는 중단됐어요`

### 실패/지연 피드백
- 오프라인
  - `지금은 연결이 없어 서버 반영이 늦을 수 있어요. 새 공유는 우선 멈췄어요`
- 서버 지연
  - `서버 반영이 조금 늦고 있어요. 잠시 후 다시 확인해주세요`

## Flow B. 전역 공유 OFF/ON

### OFF
- 목적
  - 이후 산책에서 기본 상태를 비공개로 유지한다.
- 결과
  - 다음 세션 시작 시 자동으로 공유를 켜지 않는다.

### ON
- 목적
  - 사용자가 다시 공유를 허용하되, 기본 ON처럼 오해되지 않게 한다.
- 조건
  - 회원
  - 위치 권한 허용
  - 동의 완료

### 사용자 문구
- ON 전
  - `공유는 동의 후에만 켤 수 있어요`
- ON 후
  - `이제 공유를 다시 시작할 수 있어요. 실제 반영은 산책 중에만 이루어져요`

## Flow C. 위치/알림 권한 상태 확인 및 복구

### 카드 구성
- 위치 권한 행
  - 상태: `허용됨` / `꺼짐`
  - 설명: 근처 공유와 권한의 관계
- 알림 권한 행
  - 상태: `허용됨` / `꺼짐`
  - 설명: 상태 변경/복구 안내 알림과의 관계
- CTA
  - `설정 열기`

### 상태별 문구
- 위치 권한 꺼짐
  - `위치 권한이 꺼져 있어 공유를 다시 시작할 수 없어요`
- 알림 권한 꺼짐
  - `알림이 꺼져 있어 공유 오류나 복구 안내를 놓칠 수 있어요`

### 복구 완료 후
- 앱 복귀 시 재평가
- 문구
  - `권한이 복구돼 다시 공유를 시작할 수 있어요`

## Flow D. 최근 공유 이력 / 마지막 공유 상태 확인

### 목적
- 사용자가 `최근에 공유가 언제 켜져 있었고`, `마지막으로 어떤 상태였는지` 확인하게 한다.

### 표시 항목
- 마지막 공유 시각
- 마지막 상태
  - `정상 반영`
  - `오프라인으로 보류`
  - `서버 지연`
  - `권한 부족으로 중단`
- 필요 시 최근 오류 요약 1줄

### 게스트 차이
- 게스트는 `이력 없음` 대신
  - `로그인 후 공유 상태 기록을 확인할 수 있어요`

## Flow E. 보존기간 안내 / 삭제 요청 / 관련 문서 진입

### 보존기간 안내
- 보여줄 것
  - 공유 데이터는 익명 정책 기준으로 집계/정리됨
  - 개인 식별이 아니라 상태/정책 단위로 안내
- 문구 방향
  - `공유 데이터는 정책에 따라 일정 기간 후 정리돼요`

### 삭제 요청
- entry
  - `삭제 요청`
- user expectation
  - “지금 바로 서버에서 사라진다”가 아니라 “요청이 접수되고 처리 상태를 확인한다”
- 문구
  - `삭제 요청이 접수되면 처리 상태를 알려드릴게요`

### 관련 문서
- 개인정보처리방침
- 이용약관
- 보존 정책 요약

## 오류 / 오프라인 / 서버 지연 fallback

### 오프라인
- 핵심 문구
  - `연결이 없어 현재 상태를 바로 확인할 수 없어요`
- fallback
  - 마지막 성공 상태 + 마지막 갱신 시각 표시

### 서버 지연
- 핵심 문구
  - `서버 반영이 조금 늦고 있어요`
- fallback
  - 현재 요청 상태를 badge로 분리
  - 재시도 버튼 대신 `잠시 후 자동 재확인`

### 서버 오류
- 핵심 문구
  - `지금은 상태를 불러오지 못했어요`
- fallback
  - 마지막 상태 유지 + `다시 확인`

## 회원 / 비회원 차이

### 비회원
- 상태 카드
  - `지금은 비공개예요`
  - `로그인 후 공유 상태를 관리할 수 있어요`
- CTA
  - `로그인/회원가입 열기`

### 회원
- 전체 플로우 사용 가능
- 위치 공유/근처 공유/이력/삭제 요청 모두 접근 가능

## 구현 완료 판단 기준
- 공유 OFF가 즉시 꺼진 것처럼 느껴질 피드백이 있다.
- 권한/오프라인/서버 지연 상태의 fallback 문구가 분리되어 있다.
- 최근 상태 확인과 보존/삭제 요청이 허브 안에서 독립 플로우로 이어진다.
