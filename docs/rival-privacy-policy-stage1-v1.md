# Rival Privacy Policy Stage1 v1

## 1. 목적
라이벌 기능의 익명성/노출범위/동의 흐름 정책을 제품/서버/QA가 동일 기준으로 판정할 수 있도록 고정한다.

연결 이슈:
- 정책 확정: #130
- 서버 강제 가드 구현: #150

## 2. 기본 원칙
1. 기본 비활성(opt-in): 위치 공유/익명 노출은 사용자가 명시적으로 동의하기 전까지 비활성.
2. 최소 노출: 정밀 좌표/닉네임/강아지명/개별 경로 비노출.
3. 서버 판정 우선: 앱은 요청/표현만 수행하고, 최종 노출 판정은 서버 RPC에서 수행.
4. 즉시 철회: 동의 철회 시 신규 송신/노출 즉시 중단.

## 3. 노출 판정 테이블 (Allow / Deny)
아래 조건을 모두 만족할 때만 셀을 노출한다.

| 조건 | 기준 | 불충족 시 |
|---|---|---|
| 사용자 동의 | `location_sharing_enabled = true` | Deny (완전 비노출) |
| 표본 임계 | `sample_count >= 20` | Deny 또는 percentile-only |
| 지연 반영 | 주간 30분 / 야간 60분 지연 | Deny |
| 민감 구역 | `privacy_sensitive_geo_masks` 비매칭 | Deny |
| 계정 보호 | 민감/미성년 보호 플래그 미활성 | Deny |

추가 규칙:
- `k<20` 셀은 정확 count 대신 `privacy_mode=percentile_only`로 완화 가능.
- 완화 모드에서도 정밀 좌표는 반환하지 않는다.

## 4. 식별자 정책
1. 사용자 식별자는 시즌 단위 가명(회전형)만 사용.
2. 가명은 원계정 식별자와 직접 결합되지 않도록 저장/표현 계층 분리.
3. 신고/차단 내부 처리에서는 내부 식별자 사용, 클라이언트에는 가명만 노출.

## 5. 동의/철회 상태 전이
상태:
- `OFF`: 미동의 (기본)
- `ON_PENDING`: 동의 시도 중(권한/네트워크 처리)
- `ON`: 동의 완료, 송신 가능
- `OFF_REVOKED`: 동의 철회 직후

전이 규칙:
1. `OFF -> ON_PENDING`: 사용자가 동의 버튼 탭 + 위치 권한 허용 조건 충족
2. `ON_PENDING -> ON`: 서버 저장 성공
3. `ON_PENDING -> OFF`: 실패/취소
4. `ON -> OFF_REVOKED`: 사용자가 공유 끄기
5. `OFF_REVOKED -> OFF`: 즉시 정리 작업 완료(송신 중단, UI 비노출)

일관성 요구:
- `ON` 이외 상태에서는 presence upsert 금지.
- `OFF_REVOKED` 진입 시 앱은 즉시 UI를 비노출 상태로 전환.
- 과거 데이터는 익명 집계 통계로만 유지하며 개인 식별 재노출 금지.

## 6. 개인정보 보호 리뷰 체크리스트
1. 정밀 좌표(원좌표) 응답 필드가 클라이언트 계약에 존재하지 않는다.
2. 닉네임/강아지명이 라이벌/핫스팟 응답에 포함되지 않는다.
3. 지연 반영(주간 30분/야간 60분) 규칙이 정책 테이블과 RPC에 동일 반영된다.
4. `k>=20` 임계값이 정책 테이블/문서/RPC에서 동일하다.
5. opt-out 시 신규 노출/송신 중단이 즉시 반영된다.
6. 민감 구역 마스킹 정책(`privacy_sensitive_geo_masks`)이 운영 경로에 존재한다.
7. 감사 로그(`privacy_guard_audit_logs`)로 요청별 판정 근거를 추적할 수 있다.
8. 미성년/민감 계정 보호 플래그 적용 시 강제 비노출이 동작한다.

## 7. 테스트 시나리오
1. opt-in 전/후 비교: OFF 상태에서 노출 0, ON 상태에서 정책 충족 셀만 노출.
2. 표본 경계 테스트: 19/20 경계에서 노출 모드 전환.
3. 철회 반영: ON -> OFF_REVOKED 직후 송신/노출 중단 확인.
4. 민감 구역 테스트: 마스크 영역 셀 비노출 확인.
5. 보호 계정 테스트: 미성년/민감 플래그 계정 강제 비노출 확인.

## 8. 운영/장애 대응
1. 재식별 우려 징후 시 `privacy_guard_policies`에서 지연/임계값을 상향.
2. 경보 급증 시 `view_privacy_guard_alerts_24h` 우선 점검.
3. 필요 시 feature flag로 라이벌 노출 경로를 즉시 비활성(`ff_rival_tab_v1`).
